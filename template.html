Spare Parts Master Data — Matching Strategy
SPARETECH · Data PM Case Study · May 2026
The Problem

Storeroom managers across a global manufacturing network cannot trust their inventory because the same physical part is identified differently by each site and supplier. A single Material Master export reveals the scale: of 4,203 records, over a quarter are confirmed SKU duplicates — different material numbers that map to the same physical part — and a further 591 carry placeholder article numbers that make them effectively unresolvable. The downstream effects are predictable: overstock, missed replenishment triggers, and procurement decisions made against fragmented data.

  • 4,203 total records
  • 1,111 in SKU duplicate clusters (471 groups)
  • 591 flagged placeholders
  • 2,473 clean unique records
Target Users
  • Storeroom Managers — need confidence that stocking levels and reorder signals reflect real-world inventory, not inflated counts from duplicate SKUs.
  • Procurement Teams — need accurate spend data and supplier cross-references to negotiate contracts and avoid buying parts already in stock.
  • Data Stewards — need a governed, auditable workflow for resolving duplicates without disrupting plant-level ERP views.
Strategy

No single field in the material master is reliable enough to match on alone. Manufacturer names fragment across 2–4 variants per OEM (Balluff GmbH vs Balluf vs Balluff). Article numbers carry the same part formatted differently (IE5 266 vs IE5266). Supplier fields are almost universally empty.

The solution is a tiered matching pipeline built on normalization: clean the data deterministically first, then match with confidence scores, then surface only borderline cases for human review. Every match carries an auditable rationale — not just which records matched, but exactly why. This design keeps auto-confirmed matches at 99% confidence while eliminating the need for manual review of the clear cases.

A critical structural distinction separates SKU duplicates (different material numbers = the same physical part — the real problem) from multi-plant stock (same material number at multiple locations — expected and normal). Conflating these two inflates the apparent duplicate rate and erodes trust in the matching output.

Execution Plan
  1. Phase 1 — Foundation (Weeks 1–4): Build the normalization layer (manufacturer canonicalization + article number cleaning) and run Tier 1 matching against the full Material Master. Produce the first duplicate report to quantify the problem and validate matching accuracy with a storeroom manager. This phase is complete — the output is the interactive prototype:
  2. Phase 2 — Matching Pipeline MVP (Weeks 5–10): Layer in Tiers 2 and 3 (GTIN matching, type code matching). Build a match review UI for borderline cases. Stand up a data steward queue with approve / reject / merge controls and full audit trail.
  3. Phase 3 — Supplier Cross-Reference (Weeks 11–16): Ingest supplier catalogs and map supplier article numbers to OEM article numbers. This is the core business unlock — resolving the same part across different supplier catalogs, which is where the purchasing problem actually lives.
  4. Phase 4 — Governance & Automation (Weeks 17–20): Add inbound validation rules to prevent new duplicates at the point of entry. Real-time duplicate detection fires before a new material number is created, routing likely matches to a review queue instead of creating a new SKU.
The Prototype

The applies 3 deterministic rules to all 4,203 records: (1) manufacturer name canonicalization via a lookup table, (2) article number normalization (strip spaces, hyphens, dots, fix case), and (3) Tier 1 exact matching — same canonical manufacturer + same normalized article = same physical part. Assumptions, Open Questions & Next Steps
  • Assumption: SPARETECH has or can acquire a manufacturer cross-reference database (OEM article ↔ supplier article mappings) to power Phase 3.
  • Assumption: The ERP system can surface golden record linkages as an overlay without replacing plant-local material numbers — critical for change-management.
  • Open question: The data contains two conflicting Storage Location columns, suggesting either a multi-system export or a migration artifact. Do all 7 plants share the same ERP, or are there multiple source systems with different field semantics?
  • Open question: What is the acceptable SLA for resolving a "pending review" match — hours or days? This determines staffing for the Phase 2 data steward queue.
  • Next step: Walk one storeroom manager through the 471 duplicate clusters to validate match accuracy before building the Phase 2 review UI.
  • Next step: Quantify the inventory value locked in confirmed duplicate SKUs — this is the business case number for Phase 2 investment approval.

Rule 1

Manufacturer Canonicalization

The same OEM appears under multiple name variants — typos, legal suffixes, abbreviations, mixed case. A lookup table maps every known variant to one canonical name.

Simens→Siemenstypo
Balluff GmbH→Ballufflegal suffix
TURCK→Turckcase
Hans Turck Gmbh…→Turckfull legal name
ifm electronic→ifmbrand short form
Rule 2

Article Number Normalization

The same part number is formatted inconsistently — spaces, hyphens, dots, mixed case. All are stripped to produce a consistent comparison key. Placeholders like "9999" are flagged and skipped entirely.

IE5 266→IE5266space
PCM 101210 E→PCM101210Espaces
09.20.028.0→09200280dots
1PH7107-2EG02-0BK0→1PH71072EG020BK0hyphens
9999→⚠ skippedplaceholder
Rule 3

Tier 1 Matching

After normalization, two records are flagged as the same physical part when they share the same canonical manufacturer and the same normalized article number.

An additional check separates two cases:

  • SKU Duplicate — different Material Numbers matched → the real problem
  • Multi-Plant Stock — same Material Number, multiple plants → expected, normal
